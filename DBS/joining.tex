An incoming peer ${\cal P}^?_i$ (see the tasks described in
Fig.~\ref{fig:joining}) must contact a \emph{tracker} ${\cal R}$ to
receive from it, supposing previously that in the team ${\cal T}^j$
there is room for ${\cal P}^?_i$, a \emph{reliable}\footnote{Reliable
  messages are transmitted over TCP and their transmission is denoted
  in the pseudocode by $\Rightarrow$. Unreliable messages, which can
  be lost in transit, are transmitted over UDP and its transmission is
  denoted by $\rightarrow$.} message $[{\cal S}_j]$. Then, ${\cal
  P}^?_i$ request to ${\cal S}^j$ to be included in ${\cal T}^j$. If
${\cal S}^j$ accepts, then ${\cal S}^j$ tranmits to ${\cal P}^j_i$
$[{\cal T}^j$] (more preciselly, list of peers in ${\cal T}^j$, in a
reliable message), and runs ${\cal T}^j = {\cal T}^j \cup {\cal
  P}^j_i$, in this order.

While arriving, ${\cal P}^j_i$ does not store ${\cal T}^j$ as it is
received. Instead, ${\cal P}^j_i$, for each ${\cal P}^j_k\in [{\cal
    T}^j]$, sends an (unreliable) $[\mathtt{hello}]$ message to ${\cal
  P}^j_k$, and when ${\cal P}^j_k$ receives a $[\mathtt{hello}]$, it
performs ${\cal T}^j_k = {\cal T}^j_k \cup {\cal P}^j_i$. Therefore,
the list of peers ${\cal T}^j_k$ that a peer ${\cal P}^j_k$ has,
depends in first instance, on the number of $[\mathtt{hello}]$
messages that it has received. In other words, it holds that $|{\cal T}^j_k| \leq |{\cal T}^j|$.

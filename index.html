Abstract

P2PSP is an application-layer protocol that provides real-time broadcasting of media on the Internet. Such functionaly is provided by DBS (Data Broadcasting Set of rules). The rest of sets incorporate new characteristics, when needed, buliding a highly conﬁgurable streaming system.

1 DBS (Data Broadcasting Set)

DBS provides ALM for unicast (TCP/UDP) environments. The media is received by a collection of splitters 𝒮 = {𝒮0,⋯,𝒮G−1} from a streaming server 𝒪, called source, at a (usually variable) bit-rate which matches the bit-rate of the media. Each 𝒮i splits the stream into a sequence of chunks, and relay them to diﬀerent team of up to N peers. We deﬁne the set of teams as 𝒯 = {𝒯0,⋯,𝒯G−1} and the set of peers per team Tj = {𝒫0j,⋯,𝒫N−1j}.

1.1 Joining a team


Figure 1: Peer joining.

An incoming peer 𝒫i? (see the tasks described in Fig. 1) must contact a tracker ℛ to receive from it, supposing previously that in the team 𝒯j there is room for 𝒫i?, a reliable1 message [𝒮j]. Then, 𝒫i? request to 𝒮j to be included in 𝒯j. If 𝒮j accepts, then 𝒮j tranmits to 𝒫ij [𝒯j] (more preciselly, list of peers in 𝒯j, in a reliable message), and runs 𝒯j = 𝒯j ∪𝒫ij, in this order. While arriving, 𝒫ij does not store 𝒯j as it is received. Instead, 𝒫ij, for each 𝒫kj ∈ [𝒯j], sends an (unreliable) [hello] message to 𝒫kj, and when 𝒫kj receives a [hello], it performs 𝒯kj = 𝒯kj ∪𝒫ij. Therefore, the list of peers 𝒯kj that a peer 𝒫kj has, depends in ﬁrst instance, on the number of [hello] messages that it has received. In other words, it holds that |𝒯kj|≤|𝒯j|.

1.2 Leaving a team


Figure 2: Peer leaving.

An outgoing peer 𝒫oj (see Fig. 2) from team 𝒯j must to: (1) say [goodbye] to 𝒮j and to 𝒯j (in this order), (2) relay any pending (received but yet not sent) chunks, and (3) wait for a [goodbye] from 𝒮j, which performs 𝒯j = 𝒯j ∖𝒫oj. In case of timeout, 𝒫oj resets the leaving procedure, for a number of times. When a 𝒫kj receives a [goodbye] from 𝒫oj, 𝒫kj removes 𝒫oj from its team, be running 𝒯kj = 𝒯jk ∖𝒫oj.

1.3 Chunk generation


Figure 3: Chunk generation and ﬂooding.

Each 𝒮j (see Fig. 3) divides the stream into chunks of data (chunk) of constant length C, and sends exclusively each chunk to a diﬀerent origin peer 𝒫ij, using a round-robin schema. Each chunk is enumerated with a index x, conforming the message [cx] = [x,chunk], where x = imod|𝒯j|. We deﬁne a round (in a team) as the time necessary to send two consecutive chunks from the splitter (of such team) to the same peer. This time is variable and depends on |𝒯j|, C, and the average bit-rate of the media, R.

1.4 Chunk ﬂooding

When a peer 𝒫kj receives a chunk from 𝒫ij, 𝒫kj ﬂoods the chunk to 𝒯j ∖𝒫ij, using again a round-robin schema (see Fig. 3). Besides, if it is a duplicate chunk, 𝒫kj sends to 𝒫ij a [NRFCF𝒫lj] (Not Relay Future Chunks From) message, where 𝒫lj is the origin peer of the duplicate chunk. Thus, only the ﬁrst “neighbor” 𝒫ij to send to 𝒫kj a chunk “originated” at 𝒫lj will do that in the future, at least that 𝒫k revokes this routing information by sending [RFCF𝒫lj] (Relay Future Chunks From) to one or more (possibly the rest of) peers of 𝒯kj.

1.5 Chunk buﬀering

In order to hide the jitter generated by the physical network and the protocol itself, peers need to store the received chunks in a buﬀer before playing them. A chunk cx is inserted in the position (xmodB) of the buﬀer, being B the buﬀer size in chunks and “mod” the modulo operator. A chunk is given as lost when it is time to send it to the player and the chunk has not been received.

1.6 Free-riding control at splitters

In each team there are a set of M ≤ N monitors ℳij ∈𝒯j (trusted peers whose behavior is predictable and that are only known by the splitter), which complain to their splitter 𝒮j when a chunk is lost. For every sent chunk x, the splitter performs D[x] = 𝒫kj that associate every sent chunk with its destination (origin) peer 𝒫kj. If happens that L [LCx] (Lost Chunk) messages have beeen received from any of the monitors (the splitter only considers these kind of messages if they come from a monitor) in a number of rounds (see Fig. ?? for the exact computation of the number of lost chunks per peer per round), then 𝒫kj is considered as a selﬁsh peer and it is removed from the list of peers of the splitter.

This last functionality has not been implemented, at least, as it has been explained here. The forget thread is controlled by a timmer, not by a counter of rounds.

1.7 Free-riding control at peers

In each round, peers check if a chunk have been received from the rest of peers of the team (𝒫k ∈𝒯j)). If not, peers send a [propagate𝒫i] to one or more (possibly to the rest of) peers of the team, where 𝒫i is the origin peer of the missing chunk. At this point, the process continues as described in Section 1.4.

1.8 Congestion control

DBS is a content-unaware push-based protocol. To avoid network congestion while ﬂooding, sending peers must perform some kind of data ﬂow-control. Moreover, to achieve aN ideal I/O ratio of 1, peers should send one chunk for every received one.

Congestion control in P2PSP is very simple: if a new chunk is received, peers forward (using the ﬂooding with prunning algorithm described in Sec ??) each received chunk to the next peer of their list of peers (following a round-robin pattern).

2 IMS (Ip Multicast Set)

IPM is available by default in LANs (Local Are Network)s and VLANs (Virtual LAN)s [3]. IMS runs on the top of DBS and provides eﬃcient native multicasting.

The idea of IMS is simple: all peers in the same LAN or VLAN have the same network address. When a joining peer 𝒫i receives [Sj] from ℛ, checks if there are neighbors in the same subnet (all share the same private network address). For all those peers, 𝒫i uses the IP address 224.0.0.1 (all systems on this subnet), port 1234 to forward chunks to all of them (see Procedure 1.8). When implementing IMS, all peers, request to listen to this IPM group and port.

3 LRS (Lost-chunk Recovery Set)

LRS extends DBS. LRS should be implemented if error-prone communications are expected. LRS is based on the use of monitor peers (see Procedure 1.6). The idea is: 𝒮 will resend lost chunks to one or more the monitors when one or more monitors report their loss. To increase the probability of receiving the resent chunk on time (by normal peers), monitors halves the number of chunks in their buﬀers in relation to common peers. LRS only modiﬁes the behavior of 𝒮 and the monitors. Normal peers implement DBS.

4 ACS (Adaptive Chunk-rate Set)

ACS relaxes the peer’s upload requirements imposed by DBS. It should be used in if it is known that some peers can provide the capacity than others cannot, or when we want to mix the CS and P2P models, sending more chunks from S to one or more monitors controlled by the contents provider.

ACS is based on the idea of using the information that 𝒮 knows about the number of chunks that each peer has lost (see Procedure 1.6), to send to those more reliable peers a higher number of chunks than to the others. In other words, ACS adapts the round-time of each peer to its capacity.

Notice that ACS only aﬀects the behavior of 𝒮.

5 MCS (Multi-Channel Set)

When using MDC [1], SVC [2], or for emulating the CS model, it can be interesting for peers to belong to more than one team. To implement MCS, peers must replicate the LP2PLS protocol layers for each team (channel), except the buﬀer.

The use of MDC is trivial: the higher the number received descriptions (channels), even partially, the higher the quality of the playback. However, when transmitting SVC media, peers should prioritize the reception of the most important layers. Finally, a pure CS service could be provided if the corresponding splitter announces one empty team and sends each chunk so many times as teams (with one peer/team) there are.
